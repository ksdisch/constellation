# Project audit & fix plan — 2026-07-09

Full-project audit of Constellation at `main` (`5caa420`), covering the game client, phone client, relay + protocol, deploy configs, docs, and the `.claude/` tooling. Everything below was verified against the actual code; the highest-severity findings were additionally **reproduced live** in this session (relay smoke, headless planet-1 clear, a real two-client co-op round-trip over the relay, and Phaser-source confirmation of the particle bug).

**Verdict in one paragraph:** the project is in far better shape than its docs claim — typecheck, all 156 tests, the build, a headless planet-1 clear, and a live two-client co-op round-trip all pass. The pure-logic layers (saves, talents, relay allowlist, cue/effect/track tables) are genuinely hardened and well tested. The real problems cluster in four places: **(1)** the deployed relay has two remotely triggerable process-crash vectors, **(2)** both clients are blind to their own socket dying — which will bite in the first real couch playtest — **(3)** two in-game bugs undermine shipped features (freeze re-cast *shortens* the freeze; every particle burst renders off-screen, so the whole M5 burst layer is invisible), and **(4)** the onboarding docs (README/CLAUDE.md/session-start/PROJECT_GUIDE) describe a project 8 milestones ago and actively misdirect new sessions. There is also one tooling landmine: the protocol-guard hook **auto-approves** edits to the file it exists to guard.

## Findings

Severity: **P0** = remotely triggerable breakage · **P1** = real bug that will bite in normal use · **P2** = robustness / drift that misleads or degrades · **P3** = polish, hygiene, enhancements. Effort: S < 1h, M = hours, L = a day+.

### P0 — relay process crashes (fix before the public deploy)

- **F-01 · `server/server.ts:56` vs `:62` — a 4-byte `null` frame kills the relay for everyone.** `JSON.parse('null')` succeeds inside the `try`, then `msg.type` dereferences `null` *outside* it → uncaught `TypeError` in the `'message'` listener → process exit, all rooms lost. Anyone who can reach `wss://…` can send it. Fix: guard `typeof msg?.type !== 'string'` (or move the access into the `try`). **S**
- **F-02 · `server/server.ts:49-130`, `:37`, `:47` — no `'error'` listener on connection sockets, `wss`, or `httpServer`.** `ws` emits `'error'` on the socket for invalid frames/UTF-8, over-limit payloads, and plain TCP errors (`ECONNRESET`/`ETIMEDOUT`); with no listener Node throws → process exit. A phone dropping uncleanly on cellular — the *expected* client — can produce this organically. Fix: no-op/log `'error'` handlers in the connection handler + on `httpServer`. **S**

### P1 — bugs that bite in normal use

**Game**

- **F-03 · `src/game/entities/Enemy.ts:35-43` — re-casting Freeze *truncates* the freeze.** `freeze()` schedules an unconditional un-freeze timer and never cancels the previous one: freeze at t=0 (3s), re-cast at t=2.5s (5s boosted, should hold to 7.5s) → the first timer fires at t=3 and unfreezes while the UI just told the player the sentry is safe. Phase Dash already has the token guard this needs (`Planet.ts:407-423`). Fix: store/cancel the pending `TimerEvent`. **S**
- **F-04 · `src/game/juice/effects.ts:135,146` — every particle burst renders at (2x, 2y): the entire burst layer is invisible.** The emitter GameObject is created at world `(x,y)` and `explode(count, x, y)` passes the same coords again — but Phaser ≥3.60 treats them as emitter-*local* (`Particle.fire` does `this.x += x`; verified in `node_modules/phaser/src/gameobjects/particles/Particle.js:436-486`). Every call site lands outside the 960×540 view (freeze at enemy (420,435) → (840,870); death → y≈1000; win (920,300) → (1840,600)). The bridge only asserts `lastBurst` metadata, so headless verification passes while players see nothing. Fix: `emitter.explode(spec.count)`. **S**
- **F-05 · `src/game/scenes/Planet.ts:193` + `src/game/net/client.ts:40-45` — Planet re-subscribes `net.onMessage()` every `create()` and never unsubscribes.** The unsubscribe return value is discarded; no scene hooks `shutdown`. In co-op, after N replays/planet entries one phone solve executes `castPower()` N+1 times on the live scene (stacked banners/SFX) and pushes N+1 `solveTimings` entries → the persisted M10 telemetry (`lastSolveMs`, per-power `count`) is corrupted. Solo keys and the bridge bypass the net path, so headless runs can't catch it. Fix: `const off = this.net.onMessage(…)` + `this.events.once(SHUTDOWN, off)`. **S**
- **F-06 · `src/game/scenes/Lobby.ts:59` + `src/game/net/client.ts:26-33` — Lobby's stale handler throws on destroyed Text and starves Planet of relay messages.** Same leak as F-05, worse consequence: after Lobby's Text objects are destroyed, `setText` on them throws under the WebGL renderer (`Text.updateText` dereferences the destroyed frame's `glTexture`); the net client wraps the whole handler fan-out in **one** try/catch, and Lobby registered first — so the exception silently drops that message for every later handler. Net effect in a real co-op session: the "● phone disconnected" indicator can never fire, and a rejoining phone never gets its theme re-announced. Fix: unsubscribe on scene transition + per-handler try/catch in both net clients. **S**

**Phone / relay lifecycle**

- **F-07 · `src/phone/net/client.ts:21-59` + `src/phone/App.tsx:157-184` — the phone never learns its socket died; solving a puzzle then shows a false "Cast!" and mints stardust.** No `'close'` listener exists (the `'error'` listener is removed once open) and `send()` silently no-ops when the socket isn't OPEN. Scenario: phone locks / tab backgrounded → OS kills the socket → player returns to a live-looking spellbook, solves, gets the big "Cast!" + stardust — the laptop receives nothing, indefinitely. (The reverse path works: game disconnect → relay error → join screen.) Fix: surface `'close'` via an `onClose` callback → "connection lost — rejoin" state; make `send()` return a boolean. **M**
- **F-08 · `server/server.ts` (whole file) — no heartbeat: a ghost phone permanently blocks the documented rejoin path.** A phone that sleeps on cellular never FINs; `room.phone` stays set, so a real rejoin gets `'room already has a phone'` until kernel TCP gives up (tens of minutes). Ghost *games* leak their rooms in the `rooms` map until restart. This turns CLAUDE.md's "rejoin is manual (refresh phone, re-enter code)" into "rejoin is impossible". Fix: standard `isAlive`/`pong` + 30s sweep calling `terminate()` (existing `'close'` cleanup then does the right thing); also effectively fixes unbounded-`rooms` growth (F-26). **S/M**

**Tooling / supply chain**

- **F-09 · `.claude/hooks/protocol-boundary-guard.sh:28` — the guard hook auto-APPROVES edits to the file it guards.** It emits `"permissionDecision":"allow"`, which skips the permission prompt for `src/shared/protocol.ts` edits — the opposite of the intent ("non-blocking reminder"). Fix: drop the `permissionDecision` field (keep `additionalContext`). **S** *(verified live: the JSON is exactly as described)*
- **F-10 · `package.json` — `ws@8.20.0` ships with known memory-disclosure + DoS advisories** (GHSA-58qx-3vcg-4xpx, GHSA-96hv-2xvq-fx4p; fixed in 8.21.0, in-range), plus 6 more advisories across dev deps (shell-quote critical, @babel/core, esbuild-via-tsx, undici, vite). All resolve via `npm audit fix` — no majors needed. **S**
- **F-11 · `scripts/smoke-relay.ts:54-59,113-115,22-25` — `smoke:relay` orphans its relay child, holds the port, and hangs any piped caller.** Reproduced twice this session: `relay.kill()` signals the `npx` wrapper, not the `tsx` grandchild, which survives holding port 3099 *and* the inherited stdio pipe — `npm run smoke:relay | tee …` (i.e., any CI) never terminates even after `✓ relay smoke passed`. Additionally `fail()`/the top-level catch call `process.exit(1)`, which skips the `finally { relay.kill() }` entirely, and the fixed 1200 ms boot sleep is racy on cold machines. Fix: spawn detached + `process.kill(-pid)` (or spawn `node_modules/.bin/tsx` directly / import the server in-process), poll `/healthz` instead of sleeping, kill before exit. **S/M**
- **F-12 · `.mcp.json` — the pinned playwright MCP is broken in exactly the cloud sessions it was vendored for.** `@playwright/mcp@latest` with no `--browser` flag wants the `chrome` channel; cloud containers ship only Playwright Chromium (`/opt/pw-browsers`). Hit live this session (`Chromium distribution 'chrome' is not found`); `/verify-planet` cannot run as shipped. Fix: add `"--browser", "chromium"` to the args. **S**

## The fix plan

Six phases, sequenced by risk → leverage. Each is independently shippable as one PR, respects the locked stack (zero new runtime deps anywhere), and names its verification gate.
